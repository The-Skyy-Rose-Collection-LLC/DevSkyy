  CROSS-PLUGIN.md                        (handoff doctrine — shared)
  plugins/
    skyyrose-market/   .claude-plugin/plugin.json  skills/  agents/
    skyyrose-design/   .claude-plugin/plugin.json  skills/  agents/
    skyyrose-core/     .claude-plugin/plugin.json  skills/  agents/

One local marketplace, 4 installable plugins. All skills/agents from installed plugins are globally invocable — so cross-plugin calling is native; the routing skills make it intentional.

Cross-plugin calling — the mechanism

Claude Code exposes every installed plugin's skills/agents globally, namespaced <plugin>:<skill>. To make handoffs intentional rather than accidental:

  1. Naming convention — skyyrose-market:, skyyrose-design:, skyyrose-core:, skyyrose-qa:. Predictable, greppable.
  2. Per-plugin router skill — each plugin ships a top-level <name>-dispatch skill whose SKILL.md has a "When to hand off" table pointing at sibling plugins (the "Calls →" rows above).
  3. Shared CROSS-PLUGIN.md — one doctrine doc at the marketplace root, referenced by every plugin's README, defining the handoff graph: market→design→qa→core→qa.
There is no hard plugin-to-plugin API in Claude Code — "calling" = Claude invoking a sibling plugin's named skill/agent. The router skills + naming + doctrine make that behavior reliable and documented, which is the real deliverable.

Migration mechanics

  • Restructure skyyrose-elite/ → skyyrose-suite/ with 4 plugins/<name>/ subdirs (marketplace.json lists all 4).
  • Current 36 skills + 7 agents redistribute: 36 skills + 6 agents → market; photography-director → design.
  • Copy general skills from ~/.claude/skills / .claude/skills into the right plugin (snapshots; general copies stay canonical).
  • Embed 17 repo agents (.claude/agents) across the 4 per the tables.
  • Update installed_plugins.json + known_marketplaces.json; reinstall the 4 plugins (local marketplace).

Totals: ~144 skills + 24 agents across 4 plugins. Coherent systems (gsd, per-language) stay OUT — they're not SkyyRose-ops.
