Overview

The self-healing-theme-loop is a scheduled, autonomous agent workflow that monitors the live SkyyRose storefront (https://skyyrose.co), detects regressions in HTTP health, brand canon, and asset integrity, heals confirmed regressions in an isolated git worktree, and learns from every cycle so that recurrent failure classes are progressively eliminated rather than repeatedly patched.

The loop runs on a 6-hour cron by default (configurable at launch). It is read-only and dry-run by default — no deploy occurs unless AUTO_DEPLOY=true is explicitly set by the operator.

Design goals

  • Self-healing: restore broken features automatically, gated behind php + phpcs + Playwright.
  • Self-improving: every fix must be a net improvement — root-cause removal, not symptom patches.
  • Learning: a persistent knowledge store tracks regression signatures, proven fix patterns, and prevention guards so recurrences trend downward across cycles.
  • Deterministic: workflow scripts never call time() or random APIs. All timestamps are produced by agents via shell (date +%s for cache-busting, date -u +%FT%TZ for log stamps). Per-item variation is driven by index, not randomness.
Escalation bound: if a regression is not healed after 2 consecutive cycles, the loop stops attempting it, logs the escalation, and surfaces it for the founder. It never loops forever on an unresolvable regression.

Heal + Improve — the four obligations (a–d)

The theme-heal-doctor agent must satisfy all four obligations in every healing pass. A fix that satisfies (a) but skips (b) through (d) is sent back once for a root-cause pass before being accepted.

  1. Restore the broken feature — verified against live HTML (cache-busted curl) and source.
  2. Fix the root cause, not the symptom — remove the bug class so it cannot recur in the same form.
  3. Leave touched code better — align to existing patterns, dedup, add a guard / early-return / assert, tighten escaping, remove dead references. Scope is bounded to the touched surface; no sprawling refactors.
  4. Edit source CSS/JS only, then rebuild .min — run node scripts/build-css.js && node scripts/build-js.js. Prove-absent before deleting. No innerHTML. Escape all PHP output via esc_html() / esc_attr() / esc_url().

Determinism contract

The workflow script (self-healing-theme-loop.js) is deterministic — it contains no calls to Date.now(), Math.random(), or any non-deterministic API. All timestamps are produced by agents via shell:

  • Cache-busting: date +%s
  • Log stamps: date -u +%FT%TZ

Per-item work variation is driven by array index, not randomness.

Monitor Signals (S1–S4)

All S1–S3 signals are curl-reliable — they work headless with no MCP dependency. S4 is best-effort MCP; the loop never fails a cycle on S4 alone.

S2 Canon-drift grep on cache-busted HTML yes — high
S3 Asset version curl (headless) yes — high

S3 — Asset Version

The homepage HTML must reference the current asset version matching the committed SKYYROSE_VERSION constant (baseline: 1.5.25). A minimum of 2–3 .min assets must return HTTP 200.

  • Theme assets carry ?ver=1.5.25 (WordPress core/plugin assets may carry ?ver=1.0.0 — this is not a failure).
  • Checked assets: main.min.css, design-tokens.min.css, premium-interactions.min.js.

S4 — Lighthouse + Console Errors (best-effort)

Invoked via mcp__chrome-devtools when available in the headless environment. Reports Lighthouse performance and accessibility scores, plus console errors. S4 failures are advisory only — the loop never fails a cycle, blocks a gate, or triggers a deploy on S4 signals alone. S4 data is logged to tasks/heal-log.jsonl as a medium-severity annotation. If S4 is unavailable, the cycle logs "s4": "skipped - MCP unavailable" and continues.

Self-improvement metric

The primary metric is recurrence trend: across cycles, the mean recurrences value per active signature should trend downward as preventions are added. A flat or rising trend signals that fixes are treating symptoms, not root causes.

metric_recurrence_trend in heal-knowledge.json is an array of { cycle, meanRecurrences, signaturesActive } snapshots written at the end of each cycle.

Mine step — escalation at recurrences ≥ 3

At the end of Learn-After, the loop scans all signatures with recurrences >= 3. For each:

  1. Propose a permanent structural fix (not a per-cycle patch).
  2. Propose a new monitor signal (S1–S3) that would catch this class earlier.
  3. Log both proposals to the founder via dryManifest in tasks/heal-log.jsonl.
  4. Set preventionAdded on the signature record once a proposal is accepted and implemented.

ML upgrade path (document only — do not build)

Once heal-knowledge.json and heal-log.jsonl accumulate sufficient cycles (target: ≥ 20 cycles, ≥ 5 unique signature classes), a lightweight model can be trained offline on the logged outcomes to:

  • Score regression-risk per surface (e.g., which PHP files are highest-risk post-deploy).
  • Rank fix-patterns by historical success rate per signature class.
  • Predict which S-signal will fire next given recent deployment history.

The JSON schemas are model-ready from day one: flat, typed, no nested objects beyond depth 2, ISO 8601 timestamps, and enum-constrained categorical fields (lastOutcome, gate). No schema migration is expected before ML training begins.

Heal vs Improve — Two-Axis Review Gate

After the theme-heal-doctor agent produces a candidate fix in the worktree, a code-review gate evaluates the fix on two independent axes. Both must pass.

Axis Question Evidence required
Does it heal? Does the fix resolve the detected regression? Cache-busted curl confirms the S1/S2/S3 condition that triggered the regression is no longer violated. Source diff confirms the broken code path is addressed.
Is it a net improvement? Does the fix leave the touched surface in better shape than it found it? At least one of: root cause removed (not just symptom patched), deduplication, a new guard / early-return / assert added, tighter escaping, dead references removed, alignment with existing theme patterns.
Patch-only fails axis 2. A fix that restores the broken feature but only patches the symptom (e.g., adds a try/catch without removing the cause) is sent back to the heal-doctor agent once for a root-cause pass. If the root-cause pass also fails axis 2, the fix is accepted on axis 1 alone, the regression is logged as lastOutcome: "healed" with a note, and the signature's recurrences counter is incremented to flag it for the mine step.

Rubric applied by the reviewer

  • Root cause removed: the code path that allowed the regression cannot reproduce the same fault without a new code change.
  • No innerHTML: all DOM construction via createElement + textContent or DOMParser.
  • PHP output escaped: esc_html(), esc_attr(), esc_url(), wp_kses_post() on every output point touched by the fix.
  • Source + .min in sync: both CSS/JS source and rebuilt .min files are committed together.
  • PHPCS clean: no new WPCS errors on touched PHP files.
  • Bounded scope: diff touches only the surface relevant to the regression, plus any directly related dead code. No exploratory refactors.

Safety

Read-only by default

The loop performs only read operations (curl, grep, Read tool) until a regression is detected and gate passes. No file is written to the production theme path within a cycle; all healing work happens in an isolated git worktree branched from HEAD.

Worktree isolation

The theme-heal-doctor agent creates a git worktree at /tmp/theme-heal-<cycle>/ for each healing attempt. The worktree is removed after the cycle regardless of outcome. The main branch is never modified directly by the agent.

Deploy off by default

AUTO_DEPLOY defaults to false. In dry-run mode the loop:

  1. Runs all phases through gate.
  2. Writes the deploy manifest (dryManifest) to tasks/heal-log.jsonl.
  3. Surfaces the manifest for the founder to review.
  4. Does not invoke deploy-theme.sh.
When AUTO_DEPLOY=true, deploy requires all of: gate = green, S4-only violations excluded, and STOPSHOW_ACK=1 accepted by the settings permission layer. The loop never self-grants this permission.

Escalate after 2 consecutive failing cycles

If the same regression signature is not healed after 2 consecutive cycles:

  • Set lastOutcome: "escalated" on the knowledge record.
  • Stop attempting to heal it (do not loop forever).
  • Append an escalation entry to tasks/heal-log.jsonl.
  • Add one line to .wolf/memory.md for session-level visibility.
  • Surface to the founder for manual intervention.

No self-grant

The loop never modifies .claude/settings.json, never adds itself to allowedTools, and never bypasses the STOPSHOW_ACK guard. All permission changes are made by the operator explicitly.

Bounded per-cycle budget

Each cycle is bounded: maximum one healing attempt per regression signature per cycle. The total number of healing sub-agent invocations per cycle is capped at the number of distinct regression signatures detected (never fan-out beyond that). S4-only findings produce zero healing attempts.

Never deploy on S4-only

If the only regressions detected are S4 advisory signals (Lighthouse perf, console errors), the cycle logs the findings and exits without healing or deploying. S4 findings are surfaced to the founder as advisory notes in the next dry manifest.

Runbook

Enable auto-deploy

By default the loop runs in dry-run mode. To enable live deploys, the operator must do both of the following — neither alone is sufficient:

  1. Add the deploy-theme.sh permission line to .claude/settings.json under the project's allowedTools (or equivalent bash permission entry): 
    // In .claude/settings.json → allowedTools (or bash allow list)
"bash(scripts/deploy-theme.sh:*)"
  2. Set the environment variable AUTO_DEPLOY=true when launching the cron invocation: 
    AUTO_DEPLOY=true claude run self-healing-theme-loop
STOPSHOW_ACK=1 is required. The deploy script enforces the project-wide STOP-AND-SHOW guard. The workflow engine must supply STOPSHOW_ACK=1 as a prefix when invoking scripts/deploy-theme.sh: STOPSHOW_ACK=1 bash /Users/theceo/DevSkyy/scripts/deploy-theme.sh

Cron cadence

Default: every 6 hours. To adjust, modify the cron schedule at registration time. The schedule is configurable per-environment; the workflow script does not hardcode it. Example crontab entry for every 6 hours: 

0 */6 * * *   AUTO_DEPLOY=false claude run self-healing-theme-loop >> /var/log/theme-heal.log 2>&1

Dry-run inspection

In dry-run mode (default), review the last cycle's output: 

tail -n 1 /Users/theceo/DevSkyy/tasks/heal-log.jsonl | python3 -m json.tool

Knowledge store inspection

cat /Users/theceo/DevSkyy/.claude/state/heal-knowledge.json | python3 -m json.tool

Baseline update

If the site legitimately changes (new pages, version bump), re-run the baseline-seeding agent and commit the updated theme-health-baseline.json. Do not manually edit size floors — always re-derive from a live cache-busted capture at 85% of observed.

Escalation response

When a regression appears in the heal log with "lastOutcome":"escalated", it requires manual investigation. Steps:

  1. Read the rootCause and fixPattern fields on the knowledge record.
  2. Inspect the last 2–3 cycle records for that signature in tasks/heal-log.jsonl.
  3. Apply a manual fix via the normal dev-team workflow (skyyrose-dev-team.js).
  4. After the manual fix is deployed and confirmed, reset recurrences to 0 and clear lastOutcome to "healed" on the knowledge record.

Design Decisions

Decision 1 — curl-core, not MCP-core

Choice: S1, S2, and S3 are implemented with plain curl and grep. Only S4 uses the chrome-devtools MCP.

Rationale: The loop must run reliably in headless, scheduled, and CI environments where MCP servers may not be available. curl is available everywhere. Making the loop's correctness signals MCP-dependent would make every MCP outage a missed regression. S4 is advisory precisely because it cannot be made curl-reliable.

Trade-off accepted: S4 (Lighthouse perf, console errors) is lower-fidelity in headless mode. This is acceptable because S4 regressions are medium-severity and do not gate deploys.

Decision 2 — deploy hard-gated, off by default

Choice: AUTO_DEPLOY=false is the default. Three conditions must all be true before a live deploy fires: gate = green, S4-only violations excluded, and STOPSHOW_ACK=1 present.

Rationale: The project's STOP-AND-SHOW protocol (CLAUDE.md) explicitly designates any execution of deploy-theme.sh on skyyrose.co as a production-site action requiring explicit authorization. An autonomous loop must not self-grant this. Deploying on S4-only is forbidden because Lighthouse regressions are advisory and not worth the site-touch risk.

Trade-off accepted: In dry-run mode, healed fixes sit in a worktree until the founder reviews the manifest and triggers a manual deploy. This adds latency but preserves the production safety contract.

Decision 3 — learning loop now, ML model later

Choice: The learning mechanism is a deterministic JSON store (heal-knowledge.json) with hand-written upsert logic. No ML model is built in this phase.

Rationale: The site does not yet have enough regression history to train a meaningful model. The JSON schemas are designed model-ready (flat, typed, enum constraints) so no migration is needed when the ML upgrade path is activated. The recurrence-trend metric gives the operator a human-readable signal that the system is improving without needing a model to interpret it.

Upgrade trigger: Activate the ML path after ≥ 20 cycles and ≥ 5 unique signature classes are logged. The training set is heal-log.jsonl + heal-knowledge.json signatures with lastOutcome labels.
