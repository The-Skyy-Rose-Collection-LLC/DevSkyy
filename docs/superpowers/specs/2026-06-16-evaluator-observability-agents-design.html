1. Overview & goals

DevSkyy runs ~75–80 agent classes (8 orchestrated CoreAgent domains in agents/core/factory.py:22-31 plus ~40–50 sub-agents and SDK/AOS tiers). Two cross-cutting capabilities are missing:

  1. Evaluator / Critic — automated quality gate on agent-produced content/copy before a human reviewer sees the draft. Closes the "mock-driven confidence" loop: today the content path (agents/skyyrose_content_agent.py:271 → publish_to_wordpress() at :692) creates WordPress drafts with no automated evaluation; the only scoring is manual content_feedback_scores recorded after the fact.
  2. Observability — consume per-execution metrics, detect anomalies (token spend, latency, retry storms, dead/looping agents, runaway cost), and alert. Today metrics go into the void: the AgentExecution table (agents/models.py:201) is defined and migrated but never written to — nothing constructs an AgentExecution(...) row anywhere in the codebase; _record_success/_record_failure (agents/core/base.py:475,487) are in-memory circuit-breaker state only (confirmed, obs #16055).

Design principles (from the brainstorm + research pass)

  • Framework-invoked interceptors, not routed agents. You don't "route a task" to a watcher; it would never fire. Both hang off the existing execution wrap.
  • External-feedback correction. The evaluator's revise loop is driven by a deterministic fact-gate + a separate judge — the validated path. (Intrinsic self-correction is unreliable; ours is not intrinsic.)
  • Deterministic before probabilistic. Code-based fact checks (catalog-grounded) run free and first; the LLM judge only scores taste on drafts that already passed the facts.
  • Alert, never auto-act. The observer recommends; the orchestrator or a human pulls the trigger. Honors STOP-AND-SHOW.
  • Adopt standards. Metric fields use OpenTelemetry GenAI semantic-convention names; scoring uses a 0–5 scale with chain-of-thought (both research-backed over the originally-proposed bespoke choices).

Non-goals (v1)

  • No auto-remediation (disabling/killing agents) — recommendation only.
  • No SuperAgent-tier (EnhancedSuperAgent) instrumentation — v1 covers the orchestrated CoreAgent tier. phase 2
  • No full OTel span emission / distributed trace hierarchy — v1 is a flat metrics table for alerting + cost. phase 2
  • No frontend dashboard build — v1 exposes alerts via API + EventBus; the devskyy.app admin panel is a thin follow-on. phase 2

2. Shared interceptor seam

Both agents hang off one wrap point: CoreAgent.execute_safe(task, **kwargs) (agents/core/base.py:670), which already brackets every orchestrated execution and calls the in-memory record hooks (_record_success at :685,696; _record_failure at :699,707). A single new hook runs after the inner execute() returns.

async def execute_safe(self, task, **kwargs):
    started = monotonic()
    try:
        result = await self.execute(task, **kwargs)   # existing
        status = "completed"
    except Exception as e:
        status = "failed"; result = self._error_envelope(e)
    # ── QUALITY HOOK (new, thin; no-op when disabled) ──────────────
    result = await quality_hooks.after_execute(
        agent=self, task=task, result=result, status=status,
        duration_ms=int((monotonic()-started)*1000), kwargs=kwargs,
    )
    # ───────────────────────────────────────────────────────────────
    return result

quality_hooks.after_execute() (new, agents/quality/hooks.py) does, in order:

  1. Observer.record(...) — always: build an AgentMetrics, persist one AgentExecution row, publish an AgentExecEvent on the EventBus.
  2. Evaluator.gate(...) — only when result.output_type ∈ {content, copy}: run the two-pass evaluation + bounded revise loop, returning a possibly-revised result.
The hook is a no-op unless enabled in config, so it ships dormant and is switched on per environment. Keeping the hook body in agents/quality/ (not inline in base.py) keeps the base class clean and the interceptors independently testable.

Coverage note: content flows through ContentCoreAgent (agents/core/content/agent.py:21) even when it delegates to the SkyyRoseContentAgent SuperAgent, so the gate catches orchestrated content. Direct SkyyRoseContentAgent calls that bypass the CoreAgent are not gated in v1 — a named limitation, addressed in phase 2.

3. Evaluator agent

Package: agents/quality/ (new; requires __init__.py per the project's no-implicit-namespace-packages rule).

3.1 Two-pass scoring

PassModuleWhat it doesCost
Pass 1 — deterministic gate agents/quality/deterministic_gate.py Pure Python, no LLM. Hard-fails (short-circuits to revise) on any of:
  • Product-fact mismatch vs the canonical catalog CSV (wordpress-theme/skyyrose-flagship/data/skyyrose-catalog.csv, 33 SKUs): product name, price, collection, sizes, edition size. Resolves by SKU/name; the only un-hallucinatable check.
  • Placeholder/stub markers: TODO, FIXME, lorem, {{, empty required fields.
  • Banned-phrase / canon violations: per-collection canon (e.g. "bloodline that raised me" is Love Hurts only) sourced from docs/brand/collection-stories.md; banned cross-collection quote leakage; tagline must be exactly "Luxury Grows from Concrete."
  • Required-structure check: presence of expected sections for the content type (e.g. PDP copy must have title + body; SEO copy must have meta description within length bounds).
 $0
Pass 2 — Claude judge agents/quality/judge.py Runs only if Pass 1 clears. Criterion-separated: each rubric dimension is its own evaluation with a chain-of-thought trace before the score. Dimensions: brand-voice alignment, clarity, persuasiveness, distinctiveness (anti-homogenization). Scored 0–5 integer (research: 0–5 maximizes human alignment; 0–1 float is false precision). Sonnet by default; Opus for hero/landing copy (per model-routing policy). 1 model call/draft/attempt

3.2 Scoring scale & routing

per-dimension 0–5 integer:
  1 = hard fail (must not reach reviewer)
  2 = weak (significant revision needed)
  3 = acceptable (minor notes)
  4 = good (reviewer may approve in one read)
  5 = excellent (fast-track)

composite = weighted mean of dimension scores  (weights in EvaluatorConfig)
route:
  composite < 3            → revise loop
  3 ≤ composite < 5        → surface to human reviewer + attach CoT critiques
  composite == 5           → surface to fast-track queue
  normalized = (composite - 1) / 4   # 0–1 form for any downstream system

3.3 Self/family-preference bias mitigations

A Claude judge rates Claude-written copy higher via perplexity-preference (real, unquantified for our exact case). Single-Claude judge is kept for cost, with mandatory mitigations:

  • Blind authorship: the judge prompt never states the draft is AI-written or names the generating agent.
  • Concrete grounding: brand-voice criteria reference specific locked phrases, canonical copy exemplars, and named banned patterns — never abstract adjectives ("luxurious", "authentic").
  • Anti-verbosity clause in every judge prompt: "Length is not a quality signal; credit density over word count."
  • Borderline position-bias check: when a dimension score is within 1 point of the pass threshold, re-run that dimension with rubric exemplars in reversed order; if the two scores diverge by >1, flag for human instead of auto-deciding.
  • Escape hatch: EvaluatorConfig.judge_provider lets the brand-voice dimension route to a non-Claude judge if calibration later shows inflation.

3.4 Calibration gate phase-0 prerequisite

The judge is not a hard gate until validated against humans:

  • Build a ground-truth set of ~100 brand-team-labeled drafts spanning the score range (deliberately bad → excellent), stored under tests/fixtures/evaluator_ground_truth/.
  • Compute Cohen's κ (judge vs human). κ ≥ 0.65 → judge runs as a hard gate (auto-revise on fail). κ < 0.65 → judge runs as a soft signal (annotate only, never blocks) until the rubric is revised and re-validated.
  • Monthly re-validation on 20–30 production samples; the Observer watches the live score distribution for leniency drift (§4).

3.5 Bounded revise loop

Module: agents/quality/revise_loop.py. On a failing composite (<3) or any Pass-1 hard fail:

  1. Build a specific, decomposed critique (format below) and route it back to the originating agent via the existing CoreAgent.delegate() path (agents/core/base.py:558).
  2. Regenerate → re-score.
  3. Cap = 2 attempts (config). Early-exit: if attempt 1 clears Pass 1 AND composite ≥ threshold, stop — do not consume attempt 2.
  4. If still failing after the cap, surface the best-scoring attempt, flagged, with the critique attached for the human reviewer. Preserve the revision-1 candidate for comparison on creative dimensions (attractor-state guard).

Critique format (research-driven — specificity is the highest-leverage variable)

{
  "fact_corrections": [          # from Pass 1 — comply, do not balance
    {"check": "price", "sku": "br-006",
     "found": "$245", "canonical": "$265",
     "fragment": "...priced at $245..."}
  ],
  "style_notes": [               # from Pass 2 — balance against brand voice
    {"dimension": "brand-voice", "score": 2,
     "issue": "reads generic-luxury; not Oakland-anchored",
     "fragment": "elevated essentials for the modern wardrobe",
     "fix_direction": "ground in concrete/Bay imagery per Black Rose canon"}
  ],
  "dossier_context": ""   # SKU-specific, anti-homogenization
}

Rules: cite the exact failing check + offending fragment; YES/NO-decomposed items; separate facts (comply) from style (balance); inject the per-SKU dossier so corrections are product-specific; no generic praise/encouragement.

3.6 Named domain risk

Aligned-model revision passes can flatten brand voice toward low-entropy "attractor states" (RLHF debiasing finding). Mitigation: the distinctiveness rubric dimension, SKU-specific critique grounding, and preserving the revision-1 candidate for creative-dimension comparison rather than blindly taking the last attempt.

4. Observability agent

Two halves: instrument the write, then consume + detect + alert.

4.1 Half A — instrument the write (the missing emit)

In quality_hooks.after_execute(), persist one AgentExecution row per execution. The current model (agents/models.py:201) has only tokens_used (combined) + duration_ms — insufficient (output tokens cost 3–5× input; combined total can't reconstruct cost). The DB write obtains an AsyncSession from the project's existing session factory in database/ (exact import to be confirmed in the plan).

4.2 Half B — consume + detect + alert

Three inputs → detection (agents/quality/detection_rules.py) → one output (AgentAlert):

  • EventBus subscriber (real-time): subscribes to AgentExecEvent via event_bus.subscribe() (core/events/event_bus.py:39; singleton at :79; events carry .event_type). Catches retry storms / failure bursts as they happen.
  • Scheduled scan (default 5 min): SQL over agent_executions for trends — spend deltas, p95 latency, dead agents, per-conversation cost.
  • Circuit-breaker state (in-memory): reads existing breaker state for looping/runaway signal.

4.4 Warm-up behavior

  • Trend rules (spend spike, latency) stay dormant until a configurable warm-up window of data exists (default 14 days). Liveness/policy rules (dead agent, retry storm, cost ceiling, TTL, breaker) work from minute one.
  • During warm-up: emit raw measurements to the dashboard/log (visibility, no alerts) so the baseline is observable.
  • Apply static floor bounds day-zero. Suppress trend alerts for 10–15 min after an agent deploy/restart (settling window).

4.5 Authority & alerting hygiene

  • Alert + recommend, never act. Each AgentAlert writes to the dashboard API + emits an EventBus event the orchestrator may react to, and carries a specific remediation string (e.g. "circuit-break agent X — looped 12×"). The Observer never disables/kills.
  • Severity tiers: page (cost ceiling, breaker open, fleet-wide dead) / ticket (spend trend, latency regression) / dashboard-only (warm-up, drift).
  • Hysteresis: alert once per open condition; resurface only on escalation. Alert state tracked per detection instance.
  • Warm-up shadow routing until a rule clears 70% precision; 30-day actionability review deletes rules below 20% conversion.

8. Testing plan

Target ≥85% (project rule). No mocks in integration tests.

  • Unit — deterministic gate (hallucinated-SKU/price draft → hard fail; clean → pass); rubric scorer with mocked Claude (0–5 parsing, CoT extraction, weighting); revise loop honors cap 2 + early-exit; each detection rule fires at threshold and stays quiet below; warm-up gating; cost computation vs pricing table; borderline position-swap re-check.
  • Integration — execute_safe wrap persists exactly one row + publishes one event; content output triggers gate; failing gate routes critique through delegate() and regenerates; scheduled scan over seeded rows emits the right AgentAlert; per-conversation cost accumulator trips on a synthetic loop. Real EventBus, real test DB (SQLite/PG).
  • Fixtures — tests/fixtures/evaluator_ground_truth/ (~100 labeled drafts for calibration κ); known-bad drafts; synthetic agent_executions rows incl. a runaway-conversation set.

10. Scope, risks, open questions

Risks (named, mitigated)

  • Judge self-bias → blind authorship + concrete grounding + calibration gate (§3.3–3.4).
  • Revision homogenization → distinctiveness dimension + SKU grounding + keep revision-1 candidate (§3.6).
  • Empty-table cold start → trend rules warm up; liveness rules immediate (§4.4).
  • Judge cost → free deterministic gate filters first; cap 2 + early-exit; per-agent disable.
  • Alert fatigue → tiers, hysteresis, shadow routing, 30-day review (§4.5).

Open questions for the plan

  1. Exact AsyncSession factory import in database/ for the hook's DB write.
  2. Where token usage is available at the execute_safe seam — does result carry per-call token counts, or must the hook read them from the LLM client layer? Determines whether v1 token metrics are exact or estimated.
  3. Dashboard alert API surface — new endpoint under api/v1/ vs reuse of an existing admin route.
  4. Who labels the ~100-draft calibration set, and when (gates evaluator hard-gate mode).
  5. Content-type detection at the seam. The gate triggers on result.output_type ∈ {content, copy}, but the orchestrator returns an untyped dict[str, Any] (no enforced envelope). The plan must decide how the hook recognizes content output — options: (a) add an output_type key to ContentCoreAgent results, (b) key on the agent's CoreAgentType being CONTENT, or (c) a registered set of content-producing agent names. Option (b) is the least invasive and recommended.
