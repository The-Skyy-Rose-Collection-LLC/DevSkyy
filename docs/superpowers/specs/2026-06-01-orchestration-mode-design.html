1. Goal & scope

Refactor the Anthropic "orchestration mode" loop theory into DevSkyy as a real, tested module. An orchestration mode is a session-level switch: when on, the agent runs every substantive request at maximum effort and fans work out to parallel subagents by default; when off, the same fan-out tool reverts to per-request opt-in. The mode is built from three documented pieces — an effort level, a mid-conversation mode reminder, and standing consent written into the fan-out tool's description — never an API flag.

The founder's steer: "I like the loop theory so refactor it to our codebase." Keep the raw-API ModeAgent loop mechanics intact; make it idiomatic DevSkyy; runs today on anthropic 0.83.0.

SDK reality (anthropic 0.83.0, Python 3.14)

  • output_config={"effort": ...} — stable OutputConfigParam.effort = Literal["low","medium","high","max"]. "xhigh" is not typed but is passed as a raw dict (bypasses the type check) and is already shipped in tournament.py.
  • thinking={"type":"adaptive"} — supported in stable 0.83.0 (thinking_config_adaptive_param.py).
  • Mid-conversation role:"system" — not in pinned 0.83.0 (MessageParam.role = Literal["user","assistant"]). But the latest anthropic-sdk-python (main) DOES type it: role = Literal["user","assistant","system"] plus a MidConversationSystemBlockParam content block (verified via Context7, 2026-06-01). So mid-conversation system messages are a real shipped feature, just newer than our pin — the SYSTEM_ROLE seam is forward-compatible, not hypothetical.
  • Message roles are trained to alternate user/assistant. A reminder as a separate user message would create consecutive user turns; the runs-today transport therefore folds the reminder into the user turn's content as a second text block.
  • Client + key: try: from config import ANTHROPIC_API_KEY / except ImportError: os.getenv("ANTHROPIC_API_KEY",""). (config is a package; there is no root config.py.)

4. Architecture

Two focused files (<800 lines each; functions <50 lines).

orchestration/orchestration_mode.py

  • SYSTEM_PROMPT — static top-level system string, never mutated (cache-prefix integrity).
  • MODE_ENTER, MODE_REFRESH, MODE_EXIT — the three reminder strings.
  • ReminderTransport — StrEnum: USER_REMINDER (default) | SYSTEM_ROLE.
  • @dataclass(frozen=True) ModeConfig — models (from model_ids), effort="xhigh", reminder_transport, max_tokens, turn caps, turns_between_refreshers.
  • ModeAgent — the async loop: set_mode, _due_system_messages, turn. Injectable AsyncAnthropic (test seam).

orchestration/orchestration_mode_tools.py

  • WORKFLOW_TOOL, BASH_TOOL, REPORT_TOOL — tool schemas (§10).
  • normalize_subtasks(raw) — accepts array | JSON-string | newline list.
  • handle_bash_block(block) + run_bash(cmd) — local bash_20250124, timeout + truncation. No sandbox.
  • run_subagent(client, prompt, config) — nested async loop, SUBAGENT_MODEL, bash + report_findings.
  • run_workflow(client, raw_subtasks, config) — gather+Semaphore fan-out, per-subagent failure isolation, dropped-overflow note.
Why two files, not one: the loop and the tool/fan-out layer have distinct responsibilities and distinct test surfaces. Splitting keeps each <400 lines, lets the tools layer be tested without instantiating ModeAgent, and matches the project's "many small files" rule.

The top-level system field never changes across turns, so the cached prefix stays intact — the entire point of the design. Reminders ride in the messages array, placed after the user turn they apply to.

6. The loop (semantics)

Faithful to the reference, adapted to async. Pseudocode for ModeAgent.turn:

async def turn(self, user_input: str) -> str:
    # reminder rides WITH the user turn (folded into its content under
    # USER_REMINDER) → alternation preserved, cached prefix untouched
    reminder_texts = self._due_reminder_texts()
    self.messages.extend(self._build_user_messages(user_input, reminder_texts))
    self._turns_since_reminder += 1

    for _ in range(self.config.max_main_turns):
        async with self._client.messages.stream(
            model=self._model,
            max_tokens=self.config.max_tokens,
            system=SYSTEM_PROMPT,                       # static for the whole session
            thinking={"type": "adaptive"},
            output_config={"effort": self.config.effort},
            tools=[WORKFLOW_TOOL, BASH_TOOL],
            messages=self.messages,
        ) as stream:
            response = await stream.get_final_message()
        self.messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "pause_turn":
            continue
        if response.stop_reason != "tool_use":
            text = "".join(b.text for b in response.content if b.type == "text")
            if response.stop_reason == "max_tokens":
                self.messages.pop()                     # drop truncated turn
                text += "\n\n(warning: response truncated at max_tokens)"
            return text

        tool_results = []
        for block in response.content:
            if block.type != "tool_use":
                continue
            if block.name == "Workflow":
                out, is_err = await run_workflow(self._client, block.input.get("subtasks", []), self.config)
            elif block.name == "bash":
                out, is_err = await handle_bash_block(block)
            else:
                out, is_err = f"unknown tool: {block.name}", True
            tool_results.append({"type": "tool_result", "tool_use_id": block.id,
                                 "content": out, "is_error": is_err})
        self.messages.append({"role": "user", "content": tool_results})
    return "(hit the main loop turn limit before finishing)"

State machine + transport (two methods)

_due_reminder_texts() -> list[str] — pure state machine, no transport:

  • EXIT pending → emit MODE_EXIT once, clear _mode_announced.
  • Mode on & not announced → emit MODE_ENTER once, reset refresh counter.
  • Mode on & _turns_since_reminder >= turns_between_refreshers → emit MODE_REFRESH, reset counter.

_build_user_messages(user_input, reminder_texts) -> list[dict] — attaches reminders per transport:

  • No reminders → [{"role":"user","content":user_input}].
  • USER_REMINDER → ONE user turn whose content is a list of text blocks: the input first, then one <system-reminder>-wrapped block per reminder. No consecutive same-role turns.
  • SYSTEM_ROLE → the user turn followed by standalone {"role":"system","content":text} messages (outside user/assistant alternation).

7. Concurrency model

Fan-out follows the repo idiom (director.py:603, round_table.py:1323), not the doc's ThreadPoolExecutor:

sem = asyncio.Semaphore(config.max_parallel_agents)
async def _bounded(prompt):
    async with sem:
        return await run_subagent(client, prompt, config)
results = await asyncio.gather(*(_bounded(p) for p in subtasks), return_exceptions=True)
# isolation: any Exception → "(subagent failed: : )" string, never raises

Overflow beyond max_parallel_agents is logged via structlog and noted in the joined output ("N subtasks not run; rerun in a follow-up Workflow call") — no silent truncation.

10. Tool schemas

  • WORKFLOW_TOOL — {name, description, input_schema}; description carries the verbatim Opt-in, Quality patterns, and Standing consent paragraphs from the reference. input_schema: {subtasks: array<string>} required.
  • BASH_TOOL — {"type": "bash_20250124", "name": "bash"} (Anthropic-defined, executed locally).
  • REPORT_TOOL — report_findings: {summary: string, findings: array<{claim, evidence, severity∈[high,medium,low,info]}>} required; ends a subagent's task.

14. Out of scope (YAGNI)

  • No wiring into skyyrose/multi_agent (CLI paradigm) or any existing orchestrator.
  • No hexagonal AI-port abstraction — constructor injection of the client is the entire seam. This is a focused capability module, not a vendor-swappable subsystem.
  • No real MCP / external subagent backend; subagents are self-contained raw-API loops.
  • No persistent shell session for bash (each call is a fresh subshell — documented limitation).
  • No role:"system" transport until next-opus + a newer SDK; the seam is present, the path is dormant.
