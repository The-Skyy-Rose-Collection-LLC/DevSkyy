Mission

Audit every skill and agent across all five skyyrose-suite plugins (skyyrose-core, skyyrose-design, skyyrose-market, skyyrose-qa, and the top-level skyyrose orchestrator) for the same class of gap found tonight in skyyrose-design: capability that's specified but never built, or that self-grades its own success with nothing independently checking it. Where a real gap is found, close it — following the exact pattern used tonight, not a generic rewrite.

This is not a request to make every plugin look like skyyrose-design. Marketing content, backend Python, and QA processes don't have the same "regression on a live URL" shape that theme work does. Adapt the underlying discipline (verify before claim, learn from repeated failure, independent check before "done") to what each domain actually needs — don't force-fit theme-heal-doctor's exact schema onto content calendars.

Precedent — what happened tonight, concretely

A 3D mascot mesh had a defect that resisted three independent single-shot fix attempts. Building a general adversarial-verification pattern (independent verifier, skeptical default, re-derived evidence, structured verdict — not the builder grading its own work) and running it via a Workflow (parallel fix hypotheses → independent verify each → synthesis) produced an honest result: none of the three approaches worked, but all three independently converged on the same precise, evidence-backed root cause, with a specific recommended next technique. That's a complete, valuable cycle — not a failure.

Trying to generalize this into the "elite web team" led to actually reading skyyrose-design/agents/theme-heal-doctor.md and the spec it references (docs/superpowers/specs/self-healing-theme-loop.html) rather than assuming it worked. That surfaced the real gap:

  • The spec is a mature, detailed, well-designed contract (monitor signals, learning-store schema, two-axis review gate, escalation rules) authored 2026-05-29.
  • The engine it describes (self-healing-theme-loop.js) was never built. Neither were its state files (heal-knowledge.json, theme-health-baseline.json) or the log (tasks/heal-log.jsonl). Confirmed by direct filesystem check, not inference.
  • The one agent that DOES exist (theme-heal-doctor) self-reports healed: true / isNetImprovement: true with nothing independently checking those claims — exactly the failure mode adversarial-verification exists to catch.
  • The spec's own example baseline numbers (page-size floors, ?ver=1.5.25) are stale — the live site has moved to 1.9.5, and /experience-*/ paths in the spec now 301-redirect to /collections/*/. A spec can look authoritative and still be quietly wrong about current reality — always re-verify against the live site, never trust an embedded example table at face value.

Closed part of that gap tonight:

  • skyyrose-design/agents/theme-heal-verifier.md — independent adversarial verifier for theme-heal-doctor's fixes.
  • skyyrose-design/skills/autonomous-issue-diagnose-fix/SKILL.md — generalized parallel-hypothesis escalation tier for hard issues.
  • A new #escalation-tier section appended to the existing spec documenting both, without touching the spec's existing authoritative content.
  • Real, freshly-verified baseline data (see table below) ready to seed theme-health-baseline.json once the engine itself is built.

Gap Pattern — what to actually look for

For every agent/skill you inspect, check for these specific gap types. Don't assume any of them apply — verify per-plugin, per-skill, from the actual files.

Gap typeHow to detect itHow it showed up tonight
Spec-only / dormant capability An agent or skill's description promises orchestration, monitoring, or a "loop" — grep for the file(s) it claims to depend on or be invoked by. If those files don't exist, it's dormant. self-healing-theme-loop.js referenced by the spec and by theme-heal-doctor.md's own frontmatter — never existed.
Self-graded success, no independent check An agent's own report schema includes a boolean like healed, passed, success, fixed — and nothing else in the system re-derives that claim independently. theme-heal-doctor's healed/isNetImprovement fields, un-verified before tonight.
Stale embedded reference data A spec/doc contains hardcoded example numbers, version strings, or URLs presented as current fact. Cross-check against the live/actual current state — don't trust the doc. Spec's ?ver=1.5.25 baseline vs. live 1.9.5; /experience-*/ paths vs. actual /collections/*/.
No escalation/learning loop Does repeated failure of the same task/signature get tracked anywhere, or does every invocation start from zero with no memory of what was already tried and failed? The spec's heal-knowledge.json recurrence-tracking concept — designed, never implemented.
No hard-issue tier Is there any path for "this resisted a normal fix, try something fundamentally different" — or does the system just retry the same approach or give up? Before tonight: escalate after 2 cycles meant "give up, tell the founder." Now: try the parallel-hypothesis pattern once first.

The Reusable Pattern

Two artifacts, both already built and usable as direct templates — read them, don't re-derive from scratch:

  • .claude/skills/adversarial-verification/SKILL.md (user-level, applies to any domain) — independent verifier, skeptical default, re-derived evidence, structured verdict enum, optional two-model debate loop capped at 3 rounds with forced execution on round 3.
  • skyyrose-suite/plugins/skyyrose-design/skills/autonomous-issue-diagnose-fix/SKILL.md — the escalation-tier Workflow shape (diagnose → parallel fix hypotheses → adversarial verify each → synthesis → bounded revision), adapted for WordPress/theme conventions. Use as the template for adapting to other domains — e.g. a Python backend equivalent would swap "cache-busted curl + PHPCS" for "pytest + ruff/black + mypy", a marketing-content equivalent would swap it for "brand-canon compliance check + a second reviewer agent applying the same canon docs independently."
  • skyyrose-suite/plugins/skyyrose-design/agents/theme-heal-verifier.md — concrete example of turning a self-graded agent into an independently-verified one without changing the original agent at all (verifier is a new, separate agent; the original agent's contract is untouched).
Don't force domains where the pattern doesn't fit. Not everything needs a "heal loop." skyyrose-market's ~40 social/content skills mostly produce one-shot copy/briefs with no "live regression" concept — the relevant discipline there is probably "does a second agent independently check brand-canon compliance before the copy ships" (a lighter adversarial-verification application), not a cron-monitored self-healing loop. Use judgment per-plugin; don't manufacture a monitoring loop where nothing is being monitored.

Boundaries (non-negotiable)

  • Additive only. Extend specs with new sections (as done tonight in self-healing-theme-loop.html#escalation-tier); never rewrite existing authoritative sections that other systems may already assume are stable.
  • Verify before claiming a gap. Don't assume every plugin has the same problem skyyrose-design had. Read the actual files. Some plugins (skyyrose-qa especially) may already have real infrastructure for this.
  • No hallucinated file paths. Every path in your final report must trace to an actual ls/Read from this session — the inventory above is a starting point, not a substitute for checking.
  • STOP-AND-SHOW unchanged. No deploys, no commits, no paid API calls, no production writes, without explicit confirmation — same rule as every other agent in this repo (see root CLAUDE.md).
  • Live-site checks are read-only curl/grep — safe, no confirmation needed for GET requests to public pages. Do not POST, do not authenticate, do not touch WooCommerce/media without the standard confirmation gates.
  • Log learnings as you go — .wolf/buglog.json (via python scripts/wolf_bug_id.py for the next ID, never guess one) and .wolf/cerebrum.md, per this repo's OpenWolf convention. If working in a worktree, check whether the worktree's copy is stale relative to the main repo's counter before allocating an ID (it was, tonight — this cost a false-collision risk that had to be resolved by comparing max IDs across both copies).

Expected Deliverable

  1. A per-plugin gap report: for each of the 5 plugins, which of the 5 gap types (see Gap Pattern table) actually apply, with file:line evidence — not speculation.
  2. For each REAL gap found (not every plugin will have one): a concrete, additive fix following the reusable pattern — a new verifier agent, a new escalation skill, a spec addendum section. Same quality bar as tonight: real data, no placeholders, explicit "what did NOT change" notes.
  3. An honest "no gap found here" verdict is a valid, valuable outcome for any plugin where the audit doesn't turn up a real problem — don't manufacture work to look thorough.
  4. If self-healing-theme-loop.js (the still-unbuilt engine) is in scope for this pass, note that as a separate, larger deliverable — building the actual cron-driven monitoring engine (S1–S4 checks, baseline diffing, knowledge-store read/write, cron registration via CronCreate) is real, substantial work distinct from the audit itself. Flag it, don't rush it into this pass unless explicitly asked to build it too.
