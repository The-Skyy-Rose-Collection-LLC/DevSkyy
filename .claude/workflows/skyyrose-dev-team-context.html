1. Mission & non-negotiables

  • 100% quality, no stubs. No TODO, FIXME, pass, raise NotImplementedError, placeholder values, or mock data in delivered code (test-boundary mocks only).
  • Anti-hallucination. If you haven't read it this session, you don't know it. Every claim about the codebase traces to a tool call. Read source → search → then state. Never invent file paths, function names, endpoint shapes, or exports.
  • Read before write. Read the existing file/pattern, then Edit (targeted) or Write (new files only). Match surrounding code's idiom, naming, comment density.
  • Stay in your file lane. The plan partitions files per agent. Do not edit files outside your assigned set — another agent owns them concurrently. Do NOT run repo-wide formatters/linters (make format, npm run lint --fix) during BUILD; that is the review stage's job.
  • STOP-AND-SHOW. Never call a paid API (FASHN, Gemini/GPT-Image/FLUX, paid LLM/image endpoints), deploy, write to WooCommerce/WP, upload media, or flush prod cache. Those are gated and belong to the human. You build and verify locally only.

2. Operating standard — /efficient-production

  • Files < 800 lines, functions < 50 lines, nesting < 4 levels.
  • Immutability: {...obj, key} not obj.key = val; Python @dataclass(frozen=True).
  • Error handling on every external call. Generic errors to clients; detailed logs server-side only.
  • Validate at boundaries: Zod (frontend) / Pydantic (backend) / sanitize_*+absint (WP).
  • No hardcoded secrets — env vars (.env, .env.wordpress, .env.secrets).
  • Python: line length 100, black+ruff+isort, type hints, every top-level package dir has __init__.py (no implicit namespace packages).
  • Frontend: npm not pnpm; Next 16 Cache Components → wrap every request-time reader (usePathname, useSearchParams, cookies(), headers(), use(params)) in <Suspense> or the build fails.
  • WP: extend via hooks, never modify core; REST via index.php?rest_route= not /wp-json/.
  • Reuse first. Extend an existing client/util/component before creating a new one. Three similar lines beat a wrong abstraction; a wrong abstraction beats nothing — read the codebase and decide.

4. Anti-generic mandate + driven examples

Generic output is a failure, not a draft. "Clean and minimal" is not a brief — it is the absence of one. Every surface must read as SkyyRose to someone who has never seen the brand name on the page. Below: what works (✅) and what to never ship (❌). These are the bar.

4.1 — WordPress template / PHP

❌ BAD — generic, unescaped, hardcoded, off-canon

  <h2>Black Rose</h2>                        <!-- type-rendered title: canon violation -->
  <p style="color:#C0C0C0">Shop our latest collection now!</p>  <!-- hardcoded hex + mall copy -->
  <?php echo $product->name; ?>                 <!-- unescaped output -->

6. Review-fix loop protocol (run after every checkpoint)

  1. Review the diff for the checkpoint at the /efficient-production bar: correctness bugs first, then security, then brand-canon adherence (§3/§4), then reuse/simplification.
  2. Apply fixes for every CRITICAL and HIGH finding inline. Fix MEDIUM when cheap.
  3. Re-verify: run the relevant check (php -l on touched PHP; npm run type-check for dashboard; pytest -x for Python). Quote the real output — never claim green without running it.
  4. Return: clean (no CRITICAL/HIGH left), the fixes applied, and any remaining lower-severity items.
  5. The orchestrator loops you until clean or the attempt cap. Make real progress each pass; do not re-report the same finding without fixing it.

7. WordPress health-check checklist (static — runs after every WP job)

This is the post-job sweep. Live HTTP/curl verification is the deploy step (gated, human-run) — do NOT deploy or curl production here. Static checks only:

  1. Duplicate pages. Scan inc/theme-activation-setup.php — every theme-owned slug resolves to a distinct page; no two pages share a _wp_page_template by mistake; the SKYYROSE_SETUP_VERSION gate is intact. Flag any slug that could collide (cf. DATA-01).
  2. Broken code. php -l every touched .php (use the Homebrew PHP path /opt/homebrew/bin/php). Zero parse errors.
  3. Templates not rendering. Every template slug in inc/enqueue.php's map matches an actual template-*.php filename exactly. Every get_template_part() target exists. front-page.php manually includes the shared parts (mobile-nav, cookie-consent, size-guide, toast-container) before wp_footer().
  4. PHPCS. vendor/bin/phpcs --standard=.phpcs.xml -s on touched files. Theme prefix skyyrose_ (no leading underscore).
  5. Security. No innerHTML in JS (use createElement+ textContent); all output escaped (esc_html/esc_attr/esc_url/wp_kses_post); all input sanitized; nonce + capability on every write; no hardcoded secrets.
  6. Assets exist. Every CSS/JS/image URL referenced by touched templates exists on disk. Bump SKYYROSE_VERSION when CSS/JS changed (CDN cache-bust).
  7. No regressions / no resurrection. No duplicated content sections; retired patterns stay retired (no --col-*/--lp-* aliases, no cart.js revival, no per-page reveal observers). When you remove a PHP section, remove its CSS + responsive overrides too.

Report each check as pass / warn / fail with the file:line evidence. Any fail → hand back to a fixer; the job is not done until the sweep is green.
