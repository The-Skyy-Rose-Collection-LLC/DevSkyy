SkyyRose Product Render Pipeline — OpenAI gpt-image-2 (Production Spec)

Date: 2026-06-08 · Status: blueprint (pre-build) · Replaces: nano-banana (Gemini) build at scripts/nano_banana/

1. Goal

Render every SkyyRose product as a 100%-replicated, high-fidelity, on-brand image via OpenAI gpt-image-2, using one identical, deterministic procedure so every SKU is visually consistent. Every real design element — jersey league patch, embroidery, logo — must be present. Then erase the old Gemini/nano-banana build and all old AI-produced product assets, but only after validated replacements exist.

3. The replication method (patches = method, not engine)

Plain text-to-image cannot reproduce a specific patch. The procedure feeds labeled references to the edit call:

  • IMAGE 1 — garment front (real photo / techflat) → shape, color, construction
  • IMAGE 2 — garment back or detail → secondary views
  • IMAGE 3 — collection logo close-up → exact graphic
  • IMAGE 4 — sport patch PNG (br-patch-{mlb-baseball,nba-basketball,nfl-football,hockey}.png) → exact patch, only for jersey SKUs

Prompt labels each image and instructs exact placement. Optional mask (transparent zone on IMAGE 1) localizes the patch region — note GPT-image masking is prompt-guided, not pixel-exact, so the patch reference image is the primary fidelity lever. Fallback for guaranteed-exact patch: deterministic PNG composite onto the render (Pillow) if the model's patch rendering ever drifts — keeps the rule's "100% replicated" bar absolute.

5. Hardening (production bar)

  • Env-key validation at startup — fail fast with a clear message if OPENAI_API_KEY absent. No silent fallback.
  • No silent fallback (project rule): if a required reference (garment / patch for a jersey) is missing, the SKU hard-fails and is reported — never renders an incomplete image.
  • Retry/backoff on 429 + 5xx (exponential, capped attempts); explicit timeout per call.
  • Cost cap: hard ceiling per run; abort if manifest exceeds it.
  • Dry-run mode: resolves refs + builds prompts + prints the manifest with ZERO API calls.
  • Idempotent output: deterministic filenames under renders/oai/{sku}/; never overwrites without a flag.
  • Structured logging + per-SKU metrics JSON (tokens, cost, latency, status).

6. Gating (NON-NEGOTIABLE)

Every generation is paid → STOP-AND-SHOW. No API call runs without: (1) a per-image cost manifest (SKU list × cost), and (2) explicit y. Every WC media upload of a finished render is a separate STOP-AND-SHOW. The pipeline defaults to dry-run; live generation requires an explicit flag and the gate.

7. Validation → scale → erase (sequencing)

  1. Build pipeline (free).
  2. Validation batch ~3 SKUs (a jersey w/ patch, the windbreaker that was a flatlay, one plain) → cost manifest → y → generate → founder judges fidelity.
  3. On approval → scale to all 33 (full manifest → y).
  4. Replace WC media / catalog images with new renders (gated uploads).
  5. Only then erase scripts/nano_banana/ + old AI product assets. (Do not destroy the working render path before the replacement is validated.)

8. Prerequisites / blockers

  • OPENAI_API_KEY — not found in .env / .env.secrets. Must be added before any generation.
  • gpt-image-2 account access — confirm the key's org has gpt-image-2 enabled.
  • Exact pricing — verify gpt-image-2 high-quality per-image cost before the manifest (drives the cost gate).
