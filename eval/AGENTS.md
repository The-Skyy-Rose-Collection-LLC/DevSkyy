# AGENTS.md — `eval/` (evaluation truth-doc layer)

**Scope:** All files in `eval/` — pass/fail rubrics, baselines, critique audits, design-thinking docs, simplify-rejects log, verify-impl logs, cost tracker, loop convergence stats, agent-skill inventory.

This directory is the **truth doc** for the entire build. Every gate (G1, G2, G3) reads from here. Every phase's exit criteria are checked here. Every retroactive audit reconciles against here.

---

## Allowed agents/skills

| Agent / Skill | Why |
|---------------|-----|
| **eval-harness** | Primary owner — owns the schema and structure of every eval file |

Other agents may **append** to specific eval files when their task explicitly calls for it (per phase plan). Examples:
- `verify-impl.js` writes to `eval/verify-impl/<task-id>.md` after every Step 2
- `kb-distill.js` does not write here directly but `loop-stats.js` reads KB entries to update `eval/loop-convergence.md`
- Per-page `design-thinking/<slug>.md` is written by the agent driving that page in Phase 4

## Forbidden

- **Any code agent without explicit phase assignment writing here.** `wordpress-pro` doesn't randomly add files. Each eval-file edit traces back to a phase deliverable.
- **Hand-edits to auto-generated files.** `eval/baselines.md`, `eval/cost-tracker.md`, `eval/loop-convergence.md`, `eval/agent-skill-inventory.md` are auto-maintained by scripts. Hand-edits get clobbered. Make changes in the source script.
- **Deletion of historical files.** Even if a baseline is superseded by a newer measurement, the old file stays in `eval/baselines/<YYYY-MM-DD>.md`. Truth includes history.

## Mandatory per-edit workflow

The 6-step workflow runs on `eval/` writes too — but **eval-as-meta** has a specific consideration: the verify-impl step (2) for an eval file is "does this rubric correctly capture the constraint it claims to?" not "is this rubric a known canonical pattern?" The trusted reference for an eval rubric is the source plan section it operationalizes, cited inline in the rubric.

## Schema enforcement

Every eval rubric file has a frontmatter block:

```yaml
---
name: <rubric name>
specified_by: <[v2: §N.M] | [wp: §N.M]>
phase: <0 | 0.5 | 1 | 1.5 | 2 | 3 | 4 | 5.N | 6 | 7>
test_command: <bash command that returns 0 on PASS, non-0 on FAIL>
pass_threshold: <numeric or string criterion>
last_updated: <YYYY-MM-DD>
last_updated_by: <agent name | manual>
---
```

Without the frontmatter, the file is structurally invalid and the gate that depends on it fails.
