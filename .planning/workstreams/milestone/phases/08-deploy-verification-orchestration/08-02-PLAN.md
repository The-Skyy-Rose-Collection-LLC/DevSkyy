    **Write tests first** in `tests/scripts/test_deploy_pipeline.py` following the pattern from `tests/scripts/test_deploy_theme.py`:
    - `PROJECT_ROOT = Path(__file__).resolve().parents[2]`
    - `SCRIPT_PATH = PROJECT_ROOT / "scripts" / "deploy-pipeline.sh"`
    - `run_script()` helper with `env_overrides` parameter
    - Test classes: TestHelp, TestScriptStructure, TestDryRun, TestStepNumbering, TestDependencyChecks, TestShellcheck

    For dry-run tests: The pipeline in dry-run mode will try to run `npm run build` (which IS expected to succeed since it is a local-only operation). However, in the test environment there may be no node_modules installed in the theme dir. To handle this, the test for `--dry-run` should use a `THEME_DIR_OVERRIDE` pointing to a temp dir with a minimal package.json that has a `"build": "echo build-ok"` script and an empty node_modules. OR -- simpler -- the dry-run tests should focus on source code patterns and --help output, and we validate dry-run behavior in the human checkpoint (Task 2).

    **Approach for testability**: The `--dry-run` tests should validate:
    1. Source code references deploy-theme.sh and verify-deploy.sh
    2. Source code has `--dry-run` passthrough to deploy-theme.sh
    3. Source code skips verify-deploy.sh in dry-run mode
    4. --help prints usage
    5. shellcheck passes

    **Then create** `scripts/deploy-pipeline.sh` following the research skeleton (08-RESEARCH.md "Pipeline Orchestrator Skeleton"):

    1. Header: `#!/usr/bin/env bash` with `set -euo pipefail`
    2. Path setup: SCRIPT_DIR, PROJECT_ROOT, THEME_DIR
    3. DRY_RUN=false flag
    4. Color logging (same pattern as deploy-theme.sh)
    5. `parse_args()` supporting `--dry-run`, `--help`
    6. `check_dependencies()` function: verify deploy-theme.sh and verify-deploy.sh exist and are executable
    7. `run_step()` function:
       - Takes step_num, step_name, and command as args
       - Logs `[$step_num/3] $step_name...` before execution
       - Logs success or failure after
       - Returns exit code of command
    8. `main()`:
       - Parse args
       - Print banner: `=== SkyyRose Deploy Pipeline ===`
       - Check dependencies
       - **Step 1: Build** (ALWAYS runs, even in dry-run -- local-only, catches build errors per research Pitfall 4)
         ```
         log_info "[1/3] Building theme assets..."
         (cd "$THEME_DIR" && npm run build)
         log_success "[1/3] Build complete"
         ```
       - **Step 2: Deploy** (passes --dry-run if set)
         ```
         log_info "[2/3] Deploying to production..."
         if [[ "$DRY_RUN" == "true" ]]; then
             bash "$SCRIPT_DIR/deploy-theme.sh" --dry-run
         else
             bash "$SCRIPT_DIR/deploy-theme.sh"
         fi
         log_success "[2/3] Deploy complete"
         ```
       - **Step 3: Verify** (skipped in dry-run -- nothing was deployed)
         ```
         if [[ "$DRY_RUN" == "true" ]]; then
             log_info "[3/3] [DRY RUN] Skipping verification (no deploy occurred)"
             log_info "[DRY RUN] Would verify: homepage, collections, REST API, about page"
         else
             log_info "[3/3] Verifying deployment..."
             bash "$SCRIPT_DIR/verify-deploy.sh"
             log_success "[3/3] Verification passed"
         fi
         ```
       - Print final: `=== Pipeline complete ===`

    **Critical anti-patterns to avoid** (from research):
    - Do NOT skip the build step in dry-run (it is local-only and catches errors early)
    - Do NOT run verification in dry-run (nothing was deployed, would verify old state)
    - Do NOT build inside deploy-theme.sh -- the pipeline composes separate scripts
    - Do NOT hardcode paths -- use SCRIPT_DIR-relative references

    Make script executable: `chmod +x scripts/deploy-pipeline.sh`
