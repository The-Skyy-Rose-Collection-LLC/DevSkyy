  <name>Task 1: Create verify-pre-commit.sh integration test script</name>
  <files>scripts/verify-pre-commit.sh</files>
  <action>
Create `scripts/verify-pre-commit.sh` -- a comprehensive verification script that tests each HOOK requirement by attempting commits with intentionally bad files and verifying the hook blocks them.

The script must:

1. Create a temporary git branch for testing (same pattern as verify-hooks.sh).
2. Use cleanup trap to always restore original branch state.
3. Test each requirement:

**HOOK-01 (ESLint):** Create `frontend/test-hook-eslint.tsx` with `const x: number = "not a number";` (ESLint error). Stage it. Attempt `git commit`. Verify exit code is non-zero. Clean up the file.

**HOOK-02 (Ruff):** Create `test-hook-ruff.py` at root with `import os\nimport os` (duplicate import -- ruff F811). Stage it. Attempt `git commit`. Verify exit code is non-zero. Clean up.

**HOOK-02 (Black):** Create `test-hook-black.py` at root with `x=1` (missing spaces around `=` -- black will flag this with --check). Stage it. Attempt `git commit`. Verify non-zero. Clean up.

**HOOK-02 (isort):** Create `test-hook-isort.py` at root with `import sys\nimport os` (out of order -- isort will flag). Stage it. Attempt `git commit`. Verify non-zero. Clean up.

**HOOK-03 (tsc):** Create `frontend/test-hook-tsc.ts` with `const x: number = "string";` (type error). Stage it. Attempt `git commit`. Verify non-zero. Clean up. NOTE: tsc runs whole-project, so if frontend already has type errors, this may pass for wrong reasons. The test verifies the hook runs tsc, not that this specific error is caught.

**HOOK-04 (mypy):** Create `test-hook-mypy.py` at root with `def foo(x: int) -> str:\n    return x` (return type mismatch). Stage it. Attempt `git commit`. Verify non-zero. Clean up. NOTE: mypy has 2094 pre-existing disabled errors. The test file must introduce a NEW error that mypy's current config catches.

**HOOK-05 (PHP):** Create `wordpress-theme/skyyrose-flagship/test-hook-php.php` with `<?php echo 'hi'` (missing semicolon and closing). Stage it. Attempt `git commit`. Verify non-zero. Clean up.

**HOOK-06 (pytest):** Verify that committing a Python file produces pytest output in the commit attempt. This is harder to test in isolation since tests may pass. Check that the commit output contains "Running fast unit tests" (the echo from the pre-commit hook).

**HOOK-08 (performance):** Time a commit with clean files that pass all checks. Stage one file from each language (a valid Python file, a valid TS file, a valid PHP file). Measure wall-clock time of `git commit`. Verify under 30 seconds.

Script structure:
- Use PASS/FAIL counters (same pattern as verify-hooks.sh).
- Each test: create temp file -> stage -> attempt commit -> check result -> unstage -> remove file.
- Between tests, `git reset HEAD` to unstage everything.
- After all tests, report summary.
- Exit 0 if all pass, exit 1 if any fail.

Make script executable: `chmod +x scripts/verify-pre-commit.sh`

IMPORTANT: Each test must clean up its temp files even on failure. Use per-test cleanup, not just a global trap. The global trap handles branch restoration.

IMPORTANT: For the commit attempts that SHOULD fail, use `git commit --allow-empty -m "test" 2>&1` capture pattern. The hook will cause the commit to fail, so capture output and check exit code.

Actually, `--allow-empty` won't trigger the hook on files. Use `git commit -m "test" 2>&1` after staging the bad file. The commit will fail (hook blocks it), then unstage and remove the file.

This is a more robust check than looking for a specific echo string.
