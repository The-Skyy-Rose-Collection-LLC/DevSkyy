SKYYROSE STRUCTURAL REMEDIATION — ONE-SHOT EXECUTION SPEC

Target: skyyrose.co (WordPress/WooCommerce, skyyrose-flagship theme, Elementor 4.1.4) Repo: The-Skyy-Rose-Collection-LLC/DevSkyy (monorepo — scope is the WordPress theme + WP site state only) Mode: Single-session, end-to-end execution. Understand → verify → implement → test → document. No partial deliverables. Branch: fix/structural-remediation off default. Atomic commits per workstream, git am-ready.

0. MISSION

The live site is straddling two template generations. The skyyrose-flagship shell runs the homepage and About; /collections/ and /pre-order/ still run the legacy shell ("Enter the Universe" nav, no Kids Capsule, Quick View grids, divergent footer/social handles). Every structural defect below is a symptom of that fault line plus missing redirect/data discipline. This spec closes the migration in one pass.

Six workstreams, dependency-ordered. Each has verified live evidence, implementation direction, and executable acceptance criteria. The session is DONE when scripts/structural_audit.py exits 0 against a staging/preview build and all human gates are documented in the PR.

1. NON-NEGOTIABLE GROUND RULES

  1. Source discipline. Name the verified source before acting. Phase 0 locates every file/DB object you will touch. No fishing across globs. If a target cannot be located and confirmed, log it in BLOCKED.md with what you searched and move on — do not guess.
  2. Anti-hallucination. Re-fetch the live page and reproduce each finding before fixing it. A finding that no longer reproduces is marked N/A — resolved upstream in the PR body and skipped.
  3. Brand law (hard constraints, enforced by audit script): - No blue anywhere. Blue-dominant hex = any color where B channel > R and B > G. Applies to CSS, tokens, inline styles, SVG fills. - No urgency mechanics. Countdown timers are removed, not repaired (see WS5). - No e-comm furniture: no sticky add-to-cart, no related-products rails, no urgency banners. - Collection pages are cinematic scroll-driven editorial — never product grids. The product is a moment inside the story. - Gender-neutral copy everywhere. No "menswear/womenswear/for her" framing. - Palette: void black #0A0A0A, rose gold #B76E79, blood red #8B0000, warm-biased neutrals. Type: Fraunces / Archivo / JetBrains Mono. - Oakland canon used straight, never footnoted.
  4. Two change surfaces, two mechanisms: - Theme code → edit in repo (wp-content/themes/skyyrose-flagship/ or wherever Phase 0 locates it), deploy via the repo's existing pipeline. - Site state (Elementor page JSON, WP menus, options, redirects, permalink flush) → idempotent Python scripts under scripts/remediation/ using the WP REST client pattern already established in the repo (HTTPBasicAuth + application password from .env: WP_BASE_URL, WP_USERNAME, WP_APP_PASSWORD). Every state script must support --dry-run and print a diff before writing. Never run destructive site-state scripts without --confirm.
  5. Act vs Ask gate: production WP state mutation, DNS/edge changes, and permalink structure changes are Ask (human gates, Section 8). Everything else is Act.
  6. Real validation only. The audit script runs against real HTTP responses, not fixtures.

2. PHASE 0 — DISCOVERY & VERIFICATION (do first, ~15 min)

  1. Fetch live: /, /collections/, /pre-order/, /about/, /collection-black-rose/, /experience-black-rose/, one PDP. Save raw HTML to artifacts/live-snapshot/<date>/ for before/after diffing.
  2. Reproduce and log each finding in Section 3 against the snapshot (grep assertions listed per workstream).
  3. Locate in repo and record exact paths in REMEDIATION_MAP.md: - Theme root, header.php / footer.php / template parts, functions.php, nav registration (register_nav_menus). - Which pages are Elementor documents vs theme templates (query wp/v2/pages?per_page=100&_fields=id,slug,template,meta via REST; Elementor pages carry _elementor_data meta). - Canonical catalog CSV consumed by scripts/preflight_audit.py (SKU source of truth). - Existing redirect mechanism (Redirection plugin? .htaccess? nginx? theme-level?). Record which one is live — /home/ currently 301s to /, so one mechanism already works; find it and extend it rather than adding a second. - Deployment path for theme code (CI workflow, or document manual deploy as a human gate).
  4. If the live theme code is not in the repo, pull it via REST/SFTP is out of scope — instead generate all theme changes as a patch series under patches/ with exact target paths, and mark deploy as human gate HG-4.

3. WORKSTREAMS (dependency order)

WS1 — SHELL UNIFICATION (root cause)

Evidence (verify each reproduces): - Homepage renders the identical full nav menu twice consecutively, plus three skip links ("Skip to main content", "Skip to content", "Skip to the story"). - /pre-order/ and /collections/ render the legacy header (contains "Enter the Universe", omits Kids Capsule, links Pre-Order to /preorder/) and a legacy footer with social handles theskyyrosecollection / skyyrosellc, support link /contact/#faq. - Flagship footer uses skyyroseco handles and /faq/; footer "Contact" is a mailto: while nav Contact is /contact/; "About" and "Our Story" both point to /about/.

Implementation: 1. Single canonical header/footer as theme template parts. One registered menu location (primary), one footer nav, one mobile drawer that toggles the same DOM node (aria-expanded + hidden), not a second copy. Exactly one skip link → #main. 2. Footer socials, contact email, and support URL read from a single options source (ACF options page or theme_mod) — never hardcoded. Seed with the flagship set (instagram.com/skyyroseco, x.com/skyyroseco, facebook.com/TheSkyyRoseCollection, contact corey@skyyrose.co per brand DNA). Flag the final handle choice in the PR body for Corey's 10-second confirmation (HG-1, non-blocking). 3. Strip page-level header/footer Elementor sections from /collections/ and /pre-order/ documents (site-state script: scripts/remediation/strip_legacy_shell.py, operates on _elementor_data, --dry-run prints the section IDs it will remove). 4. Canonical nav tree (one source, WP Menus): Home · Collections ▾ (Black Rose / Love Hurts / Signature / Kids Capsule) · Pre-Order · Our Story · Contact — Experiences entries removed (absorbed by WS3). Footer dedupes About/Our Story to a single "Our Story" link; footer Contact points to /contact/ (page), mailto: moves to a labeled email line.

Acceptance: - grep -c of the primary menu's first item markup on every crawled page == 1 per page. - Zero occurrences of "Enter the Universe", /preorder/, theskyyrosecollection, skyyrosellc in any page's rendered HTML. - Exactly one <nav> landmark with the primary menu; exactly one skip link.

WS2 — ROUTE ARCHITECTURE + REDIRECT LAYER

Evidence: /preorder/ referenced by legacy nav vs canonical /pre-order/; flat slugs /collection-black-rose/, /experience-black-rose/; legacy Shopify-era URLs still in Google's index now 404 on www (e.g. www.skyyrose.co/products/black-rose-crop-set); /home/ correctly 301s → / (pattern exists, extend it).

Implementation: 1. Canonical route map (LOCKED): - /collections/ → collections index (flagship shell) - /collections/{black-rose|love-hurts|signature|kids-capsule}/ → collection pages. Rewrite WooCommerce product_cat base to collections so the taxonomy archive is the collection page; cinematic layout via taxonomy-product_cat-{slug}.php overrides with products queried live. If Phase 0 shows collections are currently static pages rather than product categories, implement the routes as pages now and log the taxonomy migration as a follow-up in REMEDIATION_MAP.md — do not half-migrate. 2. 301 map (extend the live redirect mechanism found in Phase 0): - /preorder/ → /pre-order/ - /collection-{slug}/ → /collections/{slug}/ - /experience-{slug}/ → /collections/{slug}/ (post-WS3) - /experiences/ → /collections/ - Legacy Shopify era: ^/products/(.*) → best-match /product/$1, fallback /collections/; ^/collections/custom-collection.* → /collections/ - Case-normalize and trailing-slash-normalize. 3. www → apex 301 must happen at DNS/edge (host/Cloudflare level) — generate the exact rule text in REMEDIATION_MAP.md, mark HG-2 (blocking, human). 4. Emit redirects.csv (from → to → status) as the auditable artifact; the audit script tests every row live.

Acceptance: every row of redirects.csv returns a single-hop 301 to the mapped target (no chains); /collections/{slug}/ returns 200 on the flagship shell for all four collections; zero internal links to any legacy route in rendered HTML.

WS3 — MERGE EXPERIENCES INTO COLLECTIONS

Evidence: nav carries parallel 1:1 trees (Collections + Experiences: Black Rose ↔ The Garden etc.); two competing URLs per collection split link equity and force a story-vs-shop choice pre-comprehension.

Implementation: 1. The experience becomes the opening layer of each /collections/{slug}/ page: cinematic scroll-driven intro (existing experience HTML/JS migrated, not rewritten) flowing into shoppable editorial moments. Product hotspots deep-link to PDPs. No grid. 2. Performance guardrails: heavy canvas/3D mounts lazily via IntersectionObserver; prefers-reduced-motion gets a static art-directed fallback; LCP element must be static (image/text), not canvas. Budget: merged page LCP ≤ 3.0s on simulated 4G mid-tier mobile (verify with Lighthouse CI or lighthouse CLI against preview). 3. Experience routes 301 per WS2. Nav loses the Experiences tree (WS1 nav is already the post-merge tree). 4. Kids Capsule page is held to the highest standard — if its experience assets are incomplete, ship the collection page with the static editorial layer and log the experience layer as follow-up rather than shipping something diminished.

Acceptance: /experience-black-rose/ 301s to /collections/black-rose/; each collection page contains the experience intro markup and zero product-grid/Quick View components; Lighthouse LCP budget passes.

WS4 — HOMEPAGE DOM RESTRUCTURE + DATA-DRIVEN CONTENT

Evidence: in DOM order the © footer is followed by: "Made in Oakland" strip, testimonials, "Why SkyyRose", full FAQ, complete size-guide tables, cookie banner. Homepage claims Black Rose 14 / Love Hurts 5 / Signature 12 pieces while /collections/ claims "18 pieces across 3 collections" and omits Kids. Fit Guide service item "02" is the only non-linked item in that row. og:image and twitter:image point to an .mp4.

Implementation: 1. Landmark audit: exactly one header, nav, main, footer. Nothing renders after </footer> except dialog/portal markup that is hidden/aria-hidden until invoked, plus the cookie banner. 2. Testimonials + "Why SkyyRose": merge into one restrained section above the footer (the homepage currently does five pages of work — cut, don't just relocate; keep the strongest three testimonials). 3. FAQ → canonical /faq/ page with FAQPage JSON-LD; homepage FAQ section becomes a three-question teaser linking to it. Update stale copy: "ship Spring 2026" → pull the real window from Corey via PR flag (HG-3, non-blocking; placeholder: "ships in [WINDOW]" must NOT go live — if unanswered, restate as "ships on the date shown on each piece"). 4. Size guide → reusable <dialog> component (theme part), invoked from PDPs and Fit Guide card 02 (fixing the dead node). Tables render inside the dialog and on a canonical /size-guide/ page for deep links. 5. Piece counts become data-driven: single helper reading live Woo category counts (or the canonical catalog CSV at build), rendered on homepage and collections index. Hardcoded numerals removed. Collections index must include Kids Capsule. 6. og:image/twitter:image → static 1200×630 image. Generate from existing brand assets (void black field, rose-gold script logo, Fraunces wordmark) if no photographic asset is production-grade — do NOT use AI-generated garment imagery (known blocker; not launch-approved).

Acceptance: post-</footer> DOM contains only hidden dialogs + cookie banner; homepage and collections index piece counts derive from one function and agree; /faq/ returns FAQPage JSON-LD that validates; og:image URL ends in .webp|.jpg|.png and returns 200 with image MIME.

WS5 — COMMERCE LAYER HARDENING

Evidence: Featured grid uses ?add-to-cart={id} GET anchors; pre-order page shows a zeroed 00:00:00:00 countdown, an empty product grid, and a placeholder.jpg product image.

Implementation: 1. Replace all GET add-to-cart anchors with Woo AJAX add-to-cart (archive setting + wc-add-to-cart script) or POST + nonce. Add rel="nofollow" regardless. Confirm cart/checkout/account are excluded from page cache and that no cached page sets Woo session cookies. 2. Rebuild /pre-order/ on the flagship shell (WS1 gives you the shell for free): editorial layout, driven by the canonical catalog — render READY SKUs only; PENDING_USER_ASSETS SKUs (br-007, sg-009, sg-012, br-012, lh-006, sg-015) never render. Delete the countdown timer entirely — urgency mechanics are banned by brand law. Do not repair it. No placeholder images may render; a SKU without a production image does not appear. 3. Featured section on homepage: keep, but buttons become "View Piece" → PDP (editorial posture) with AJAX quick-add as secondary — no sticky cart behavior.

Acceptance: zero add-to-cart= substrings in any anchor href site-wide; /pre-order/ contains no countdown markup, no placeholder image URLs, and renders ≥1 READY SKU; fetching a product page twice shows no cart-session cookie on a cached response.

WS6 — VERIFICATION SUITE (ships with the PR)

Create scripts/structural_audit.py (stdlib + requests + beautifulsoup4): crawls a target base URL (env AUDIT_BASE_URL, defaults to production) and asserts, exiting non-zero on any failure:

  1. Shell: one nav landmark, one skip link, no legacy-shell markers (strings from WS1), no banned social handles.
  2. Routes: every redirects.csv row → single-hop 301; canonical pages → 200.
  3. DOM: nothing after </footer> except hidden dialogs/cookie banner (parse, don't regex).
  4. Data: piece counts scraped from homepage == collections index == live Woo category counts.
  5. Commerce: no add-to-cart= hrefs; no placeholder images; no countdown markup; og:image is a static image MIME.
  6. Brand: no blue-dominant hex in any linked stylesheet or inline style (reuse the RGB channel-comparison approach — B > R and B > G — already proven in the repo; do not use grep pattern lists). No gendered merch framing strings ("menswear", "womenswear", "for her", "for him").
  7. Meta: exactly one canonical link per page; no page canonicalizes to a legacy route.

Wire into CI if a workflow exists (Phase 0); otherwise document the invocation in REMEDIATION_MAP.md.
