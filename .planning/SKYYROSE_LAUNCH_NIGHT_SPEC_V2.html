C. WS7 — DASHBOARD ↔ WORDPRESS WIRING (full read+write, real endpoints)

C1. Auth decision (LOCKED): two credentials, one pattern — server-side only

All three auth systems are installed; use the right one per API and ignore OAuth2 entirely (interactive-grant overhead, wrong fit for machine-to-machine tonight):

Purpose API Auth Credential
Catalog/orders/customers R+W wc/v3/* Basic auth over HTTPS WooCommerce REST consumer key/secret (wp-admin → WooCommerce → Settings → Advanced → REST API → Add key, permission Read/Write)
WP core + custom namespaces R+W (wp/v2, skyyrose/v1/settings, agents-manager) everything non-WC Basic auth Application Password on a dedicated devskyy-service user (role: Shop Manager + capability grant for the custom routes' permission_callback; Phase 0 reads the theme source to confirm which capability those routes check — likely manage_options — and documents the minimal role)
Public reads (products for display, collections map, stock) wc/store/v1/*, skyyrose/v1/collections, skyyrose/v1/stock/{sku} none none — these are already public

JWT (jwt-auth/v1) is redundant given Application Passwords; do not wire it. HG-7 (blocking): Corey creates both credentials and drops them into Vercel env + local .env. Never in chat, never committed. The spec's scripts fail fast with a clear message if env is missing.

C3. Dashboard integration layer (Next.js repo)

Build a single typed WP client module — lib/wp/client.ts — no scattered fetches:

  1. wpFetch(path, opts) — server-only (import 'server-only'), base URL from env, auth header selected by namespace prefix (/wc/v3 → WC keys; /wc/store + public skyyrose routes → none; else app password), 2 req/s throttle + retry w/ Retry-After honor (delta A9), JSON error normalization.
  2. Typed methods over the REAL routes: - getCollections() → GET /wp-json/skyyrose/v1/collections - getProducts({category, page}) → GET /wp-json/wc/store/v1/products (public, paginated via X-WP-Total) - getStock(sku) → GET /wp-json/skyyrose/v1/stock/{sku} - get3DProducts(category) → GET /wp-json/skyyrose/v1/products/3d/{category} - getOrders({status}), updateOrder(id, patch) → wc/v3/orders (auth) - updateProduct(id, patch), batchProducts(ops) → wc/v3/products(/batch) (auth) — reuse the repo's established batch pattern - getSettings()/updateSettings(patch) → skyyrose/v1/settings (auth) - pushNarrative(payload) → POST /skyyrose/v1/settings/narrative (auth) - getAnalyticsSummary() → skyyrose/v1/analytics/summary (confirm auth requirement in Phase 0 — if public, flag as finding: should be authenticated, and add auth in theme) - agentsOpenState(get|set) → /agents-manager/open-state (auth)
  3. Route handlers (the dashboard's real API): - GET /api/health → returns {dashboard:"ok", wp:{reachable, authed_wp, authed_wc}, latency_ms} by hitting skyyrose/v1/collections (public) + a 1-item authed wc/v3/products?per_page=1 — this endpoint currently 404s in production; it becomes the wiring heartbeat. - GET /api/catalog/summary → live counts by collection + uncategorized list (powers launch-readiness widget; same numbers the WS6 audit checks). - POST /api/webhooks/woocommerce → receiver for Woo webhooks (below): verifies X-WC-Webhook-Signature HMAC-SHA256 base64 against WP_WEBHOOK_SECRET, then triggers revalidateTag('catalog')/revalidatePath for affected views. Reject unsigned/invalid with 401.
  4. Dashboard pages consume ONLY the typed client (server components / route handlers). Remove any hardcoded/mock catalog data found in the dashboard repo — Phase 0 inventories all fetch/axios calls and dead mock JSON; delete or migrate every one.
